# 1. Project Overview & Tech Stack
The Transport Management System (TMS) is a single-company platform built for one transport business. It covers the full operations lifecycle: trip planning, driver assignment, fleet management, real-time FASTag toll tracking, fuel card expense management, GST/RCM invoicing, and financial reporting — all in one dashboard with role-based access for every team member.
## 1.1 Tech Stack
| Layer | Technology | Purpose |
| Frontend | Next.js 14 (App Router) | Public site + Admin dashboard, SSR/SSG/ISR |
| Backend | Node.js + Express | REST API, business logic, third-party integrations |
| Database | MySQL 8 via Prisma ORM | All relational data, RBAC, audit trail |
| Auth | JWT + Refresh Tokens + MSG91 | OTP-based login, role-scoped sessions |
| Styling | Tailwind CSS v3 | Company-branded design system |
| Animation | Framer Motion + GSAP | Page transitions, task completion FX |
| OTP | MSG91 Flow API | Mobile OTP for all user logins |
| FASTag | NETC FASTag API / IHMCL API | Real-time toll deductions, vehicle tag management |
| Fuel Card | HPCL FleetCard API / IOCL XtraRewards API | Fuel transactions, card balance, reports |
| File Store | AWS S3 | POD images, receipts, invoice PDFs |
| PDF | Puppeteer / PDFKit | Invoice PDF generation with GST layout |
| Hosting | Vercel (Frontend) + Railway or EC2 (API) | Production deployment |

## 1.2 Core Modules

| i18n | Dashboard and public pages in English, Hindi, and Gujarati. |

# 3. User Flow & Authentication
## 3.1 OTP Login Flow (MSG91)
| Step | Action | Detail |
| 1 | User enters mobile | 10-digit mobile on /login page |
| 2 | API validates user | SELECT from users WHERE mobile = ? AND is_active = TRUE |
| 3 | MSG91 triggered | POST to api.msg91.com/api/v5/flow with flow_id + OTP variable |
| 4 | OTP stored | { mobile, msg91_request_id, expires_at: +5min } in otp_sessions |
| 5 | User enters OTP | 6-digit OTP on /verify-otp page |
| 6 | API verifies | GET msg91.com/verify?otp=X&mobile=Y — checks request_id |
| 7 | JWT issued | Access token (15 min) + Refresh token (7 days) in httpOnly cookies |
| 8 | Redirect | Next.js middleware reads role from JWT, routes to correct dashboard |

## 3.2 MSG91 Config
| Variable | Value |
| MSG91_AUTH_KEY | From MSG91 dashboard > Developers > Auth Key |
| MSG91_FLOW_ID | OTP Flow ID from MSG91 > Flows (create OTP flow with {{otp}} variable) |
| MSG91_SENDER_ID | TMSAPP (6-char) |
| MSG91_DLT_TE_ID | DLT Template Entity ID — mandatory for India |
| OTP_EXPIRY_SEC | 300 (5 minutes) |
| OTP_MAX_ATTEMPTS | 3 per mobile per 15 min, then lock 1 hour |
| RESEND_COOLDOWN | 60 seconds |

DLT Registration is mandatory for all transactional/OTP SMS in India. Register template at your telecom DLT portal before going live.

# 5. API Design — Pagination, Filters & Query Params
Base URL: https://api.yourdomain.com/v1 — All protected routes require Authorization: Bearer <jwt>. Role validated from JWT on every request via hasPermission(resource, action, scope) middleware.
## 5.1 Standard Response Envelope
{
  "success": true,
  "data": [ ...items ],
  "meta": {
    "total": 480,         // total matching records (ignoring pagination)
    "page": 2,            // current page (1-indexed)
    "limit": 25,          // items per page
    "totalPages": 20,     // Math.ceil(total / limit)
    "hasNext": true,
    "hasPrev": true
  },
  "filters_applied": { "status": ["in_transit"], "driver_id": 5 }
}
## 5.2 Universal Query Params (all list endpoints)
| Param | Type | Default | Description |
| page | integer | 1 | Page number — 1-indexed |
| limit | integer | 25 | Items per page — max 100 |
| sort_by | string | created_at | Column to sort. Validated against allowed columns whitelist |
| sort_order | string | desc | asc | desc |
| search | string | — | Full-text search across indexed fields |
| from_date | ISO date | — | created_at >= from_date |
| to_date | ISO date | — | created_at <= to_date |
| include | string | — | Comma-separated eager-load: include=client,driver,vehicle |

# 6. FASTag Integration
FASTag is mandatory for all commercial vehicles in India under NETC (National Electronic Toll Collection) programme by IHMCL/NPCI. Each FASTag is linked to a vehicle registration and pre-paid wallet. TMS integrates with FASTag issuer APIs to pull real-time toll deductions, wallet balance, and transaction history — and maps them automatically to trips.
## 6.1 Integration Architecture
| Component | Detail |
| Protocol | REST API (HTTPS) — bank/issuer-specific endpoints |
| Auth | OAuth 2.0 client credentials or API Key per issuer |
| Trigger | Cron job every 15 min to pull new transactions + balance |
| Mapping | fastag_transactions matched to trips by vehicle_id + transaction_at within trip date window |
| Auto-expense | FASTag toll transactions auto-created as trip_expenses (expense_type=toll, is_fastag_synced=TRUE) |
| Auto-approve | Toll expenses from FASTag are auto-approved (no Finance queue needed) |
| Balance alert | Low balance SMS + dashboard notification when balance < threshold |

// workers/fastagSync.ts — runs every 15 minutes
async function syncFastagTransactions() {
  const wallets = await db.fastag_wallets.findMany({ where: { is_active: true } });
  for (const wallet of wallets) {
    const lastSync = wallet.balance_synced_at;
    const txns = await fastagAPI.getTransactions(wallet.api_account_id, lastSync);
    for (const txn of txns) {
      const trip = await matchTripByVehicleAndTime(wallet.vehicle_id, txn.transaction_at);
      await db.fastag_transactions.upsert({ where: { transaction_id: txn.id }, ... });
      await db.trip_expenses.upsert({  // auto-create expense
        expense_type: "toll", amount: txn.amount, is_fastag_synced: true,
        status: "auto_approved", trip_id: trip?.id ... });
    }
    await updateWalletBalance(wallet.id);  // refresh balance
  }
}

# 8. Invoice Management & RCM (Reverse Charge Mechanism)
Transport services under GTA (Goods Transport Agency) have special GST rules under Section 9(3) of the CGST Act. RCM shifts GST liability from the transporter to the recipient.
## 8.1 When RCM Applies
| Scenario | GST Treatment | Invoice Shows |
| Recipient is GST-registered business (most B2B clients) | RCM — recipient pays GST to govt directly | No GST charged on invoice. Note: "Tax payable under Reverse Charge" |
| Recipient is unregistered individual | Forward Charge — GTA collects and pays GST | GST (5%) added to invoice |
| GTA opts for 12% with ITC | Forward Charge — GTA pays GST | GST (12%) added to invoice |
| Exempt categories (agri produce etc) | Nil rated / Exempt | No GST, note exemption |

## 8.2 GST Rate Reference (GTA)
| Rate | ITC | Who Pays | Notes |
| 5% | No ITC by GTA | RCM if recipient registered | Default GTA rate — most common |
| 12% | GTA gets ITC | Forward charge by GTA | GTA opts in by declaration |
| 0% | — | N/A | Exempt supply (specific categories) |

## 8.3 Inter-state vs Intra-state (IGST vs CGST+SGST)
// server/lib/gst.ts
function determineGSTType(companyStateCode: string, clientStateCode: string) {
  return companyStateCode === clientStateCode ? "intrastate" : "interstate";
}
function calculateGST(subtotal, rate, gstType, isRCM) {
  if (isRCM) return { igst:0, cgst:0, sgst:0 };  // RCM — no tax on invoice
  if (gstType === "interstate") return { igst: subtotal * rate/100, cgst:0, sgst:0 };
  return { igst:0, cgst: subtotal * rate/200, sgst: subtotal * rate/200 };
}
## 8.4 Invoice Calculation Flow
| Step | Calculation |
| 1. Subtotal | freight + loading + unloading + detention + other_charges |
| 2. Determine RCM | Check is_rcm flag (from client.rcm_applicable or manual override) |
| 3. Determine GST type | Compare company.state_code vs client.state_code |
| 4. Calculate GST | If RCM: all GST = 0. Else: IGST or CGST+SGST at applicable rate |
| 5. TDS (Sec 194C) | If applicable: tds_amount = total * tds_rate / 100 |
| 6. Net payable | subtotal + igst + cgst + sgst - tds_amount |
| 7. Store all components | All values stored separately for audit and GST returns |

## 8.5 Mandatory Invoice PDF Fields (GTA)
Supplier: name, address, GSTIN, PAN, state code
Invoice number (sequential per financial year: INV-2024-25-0001)
Invoice date
"Whether tax is payable on Reverse Charge: Yes / No" — mandatory field
Client: name, address, GSTIN, state code
Place of supply (client state name)
Service description: "Road transport of goods from [Origin] to [Destination]"
LR numbers covered
E-way bill numbers
Freight + other charges line-by-line
IGST / CGST / SGST breakdown (or NIL if RCM with note)
TDS deduction line (if applicable) with Section reference
Total amount in figures AND words (in Hindi if bilingual)
Bank account details (account number, IFSC, bank name)
Authorised signatory with name and designation
Digital signature (optional but recommended)
## 8.6 Invoice Number Auto-generation
function generateInvoiceNumber(financialYear: string, lastSeq: number): string {
  // financialYear: "2024-25" derived from invoice date (Apr-Mar)
  const seq = String(lastSeq + 1).padStart(4, "0");
  return `INV-${financialYear}-${seq}`;  // e.g. INV-2024-25-0042
}
Sequence resets on April 1 each year. Use DB atomic counter (SELECT FOR UPDATE) to prevent duplicate numbers under concurrent invoice creation.

## 17.2 Deployment Architecture
| Service | Platform | Notes |
| Next.js | Vercel | Auto-deploy from main, Edge middleware, ISR, image opt |
| Node.js API | Railway / AWS EC2 | Docker + PM2, auto-restart on crash |
| MySQL | PlanetScale or AWS RDS | Managed MySQL with daily backups, read replica for reports |
| Redis | Upstash or AWS ElastiCache | OTP sessions + rate limiting + cron lock |
| File Storage | AWS S3 ap-south-1 | Private bucket, pre-signed URLs for POD/receipt access |
| Cron Workers | node-cron inside API process | FASTag sync, Fuel sync, Document expiry alerts |
| DNS | Cloudflare | DDoS protection, proxy, SSL |
| Monitoring | Sentry + Axiom | Error tracking + structured log aggregation |

## 17.3 Security Checklist
Cookies: httpOnly + Secure + SameSite=Strict for JWT and refresh tokens
CORS: whitelist only frontend domain — no wildcard origins in production
Fuel card numbers: AES-256 encrypted in DB, displayed masked (****1234) in UI
FASTag tag IDs: stored plain (not sensitive), but access gated by RBAC
SQL: Prisma ORM only — zero raw SQL template literals
File uploads: MIME + extension validation, 10MB max size limit
OTP brute force: 3 attempts per mobile per 15 min, 1-hour lockout on exceed
Third-party API keys: never exposed to frontend; all calls server-side only
Audit log: immutable — no UPDATE or DELETE allowed on audit_logs table
RCM/GST calculation: server-side only — client sends amounts for display only, server recalculates
Invoice numbers: server-generated sequential with atomic DB lock — cannot be forged by client
All third-party API credentials: environment variables only, never in code or git
